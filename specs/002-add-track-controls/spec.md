## User Scenarios & Testing *(mandatory)*

### User Story 1 - 看到清晰的賽道路面 (Priority: P1)

玩家開始比賽後，畫面中央應顯示可識別的賽道路面（深色路面搭配白色邊線），讓玩家一眼就能區分賽道範圍與賽道外的草地或地面。目前賽道只有灰色矩形邊界多邊形，缺乏任何路面標示，玩家無法判斷應在哪裡行駛。

**Why this priority**: 可見賽道是賽車遊戲的核心視覺基礎。沒有清楚的賽道，玩家無法理解行駛路線，其他所有功能都失去意義。

**Independent Test**: 進入任意賽道後，截圖可辨識出明確的環形路面，與賽道外區域在顏色/材質上有明顯對比，無需任何其他說明。

**Acceptance Scenarios**:

1. **Given** 玩家選擇賽道並開始比賽，**When** 畫面載入完成，**Then** 顯示環形或封閉式的賽道路面，路面顏色明顯不同於賽道外地面。
2. **Given** 賽道已渲染，**When** 玩家沿賽道路面行駛，**Then** 賽車始終可見地行駛在路面上，視覺上清楚位於賽道範圍內。
3. **Given** 不同賽道（新手灣道、急彎山路），**When** 分別載入，**Then** 兩條賽道呈現不同的路線形狀，讓玩家感受到差異。
4. **Given** 起跑線位置，**When** 賽道渲染完成，**Then** 起跑終點線有視覺標示（例如黑白格紋），讓玩家清楚知道起跑及完成圈數的位置。

---

### User Story 2 - 使用螢幕上的油門與煞車按鈕 (Priority: P2)

玩家在行動裝置或觸控螢幕上遊玩時，能透過畫面右側的油門（▲）與煞車（▼）按鈕控制賽車速度，不需依賴實體鍵盤。

**Why this priority**: 行動裝置沒有鍵盤，缺少觸控操控就無法玩遊戲。此功能讓遊戲可在手機和平板上正常運作，大幅擴展可遊玩的裝置範圍。

**Independent Test**: 在觸控裝置上，按住油門按鈕，賽車應加速（速度計數值上升）；放開後賽車應減速；按住煞車按鈕，賽車應明顯減速。可完全不使用鍵盤完成一圈。

**Acceptance Scenarios**:

1. **Given** 賽局進行中，**When** 玩家觸碰並持續按住油門按鈕，**Then** 賽車持續加速，HUD 速度數值上升。
2. **Given** 賽車正在移動，**When** 玩家觸碰並持續按住煞車按鈕，**Then** 賽車明顯減速，HUD 速度數值下降。
3. **Given** 玩家同時使用鍵盤，**When** 鍵盤與螢幕按鈕同時按下，**Then** 兩種輸入均有效，不互相干擾。
4. **Given** 賽局暫停或結束，**When** 暫停/結束畫面顯示，**Then** 油門與煞車按鈕隱藏或停止響應。

---

### User Story 3 - 使用螢幕上的方向盤轉向 (Priority: P3)

玩家在行動裝置上遊玩時，能透過畫面左側的左轉（◄）與右轉（►）方向按鈕控制賽車轉向，不需依賴實體鍵盤。

**Why this priority**: 轉向控制與油門/煞車共同構成完整的行動裝置操控體驗；沒有方向控制，玩家仍無法遊玩。雖與 P2 同樣重要，但油門/煞車更為基礎，故排在 P3。

**Independent Test**: 在觸控裝置上，按住左轉按鈕，賽車應向左轉彎；按住右轉按鈕應向右轉彎。可完全不使用鍵盤完成一整圈。

**Acceptance Scenarios**:

1. **Given** 賽局進行中，**When** 玩家觸碰並持續按住左轉按鈕，**Then** 賽車持續向左轉向。
2. **Given** 賽局進行中，**When** 玩家觸碰並持續按住右轉按鈕，**Then** 賽車持續向右轉向。
3. **Given** 玩家同時按住左轉與右轉按鈕，**When** 兩個按鈕均持續按住，**Then** 兩個輸入相互抵銷，賽車直線行駛。
4. **Given** 螢幕操控按鈕可見，**When** 玩家在桌機以滑鼠按下按鈕，**Then** 按鈕同樣響應，讓桌機玩家也可使用滑鼠操控。

---

### Edge Cases

- 若賽道資料的邊界多邊形點數少於 3 個，賽道路面應跳過渲染並記錄警告，遊戲不應崩潰。
- 觸控操控按鈕在螢幕非常小（寬度小於 360px）時，按鈕應保持可點擊的最小尺寸，不應被截斷。
- 玩家在觸控按鈕上滑動手指（非單純按下/放開），應持續被視為按住狀態，直到手指離開按鈕範圍。
- 賽道路面與邊界多邊形的形狀不一定是矩形，需支援不規則多邊形的路面渲染。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 為每條賽道渲染封閉式路面，路面顏色與賽道外地面形成明顯對比。
- **FR-002**: 系統 MUST 在賽道起跑終點線位置渲染可識別的視覺標示（例如黑白格紋條紋）。
- **FR-003**: 不同賽道（例如「新手灣道」與「急彎山路」）MUST 呈現不同的路線幾何形狀，讓玩家感受差異。
- **FR-004**: 系統 MUST 在賽局 HUD 層疊顯示觸控操控介面，包含：左轉、右轉、油門、煞車四個按鈕。
- **FR-005**: 觸控操控按鈕 MUST 支援 `touchstart`/`touchend`（行動裝置）及 `mousedown`/`mouseup`（桌機）事件，兩種輸入均可驅動賽車。
- **FR-006**: 觸控操控輸入 MUST 與現有鍵盤輸入整合至同一個輸入狀態物件，不允許各自獨立處理。
- **FR-007**: 賽局暫停或結束時，觸控操控按鈕 MUST 隨 HUD 一同隱藏，不再響應輸入。
- **FR-008**: 在螢幕寬度小於 768px 的裝置上，觸控按鈕 MUST 大到足以用拇指點擊（每個按鈕最小點擊區域 44×44px）。

### Key Entities

- **賽道路面（TrackSurface）**：代表賽道的可行駛區域視覺層，由賽道的邊界多邊形資料衍生，包含路面顏色與起跑線標示。
- **觸控操控介面（TouchControlsUI）**：疊加於 HUD 之上的 DOM 元素組，包含四個方向／速度按鈕，將觸控/滑鼠事件對應至輸入狀態。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 玩家在進入賽道後 1 秒內，能目視辨識出清晰的環形賽道路面，無需閱讀任何說明文字。
- **SC-002**: 玩家在觸控裝置上，僅使用螢幕按鈕（不使用鍵盤），能在 5 分鐘內完成至少一圈完整的賽道。
- **SC-003**: 賽道路面在目前提供的兩條賽道（新手灣道、急彎山路）上均可正確渲染，無任何視覺錯誤或黑屏。
- **SC-004**: 觸控操控按鈕對輸入的響應延遲不超過 1 個畫面更新週期（約 16ms），玩家感受不到明顯的輸入延遲。

## Assumptions

- 賽道路面的幾何形狀以現有 `boundaryPolygon` 資料為基礎衍生；暫不另行設計獨立的路面多邊形資料結構。
- 觸控操控介面採用 DOM 按鈕方式實作（不使用 Canvas/WebGL 繪製），以確保無障礙性與事件處理的可靠性。
- 鍵盤操控不受任何影響，桌機玩家的體驗維持不變。
- 本規格不涵蓋搖桿（Gamepad API）支援，該需求可作為未來獨立功能追蹤。
